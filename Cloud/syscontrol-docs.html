1 TL;DR

For a server without a domain pointed yet (test server):

  1. Clone the repo
  2. cd into ./rubricapp/Cloud
  3. ./syscontrol.sh --fresh
  4. cd ~/RUBRICAPP_PRODUCTION/rubricapp/Cloud
  5. Set up your BackEndFlask/.env file — see Section 2
  6. ./syscontrol.sh --init — enter your domain, answer Y to continue, answer N to the SSL question
  7. ./syscontrol.sh --serve — enter your domain when prompted

For a server with a domain already pointed:

  1. Clone the repo
  2. cd into ./rubricapp/Cloud
  3. ./syscontrol.sh --fresh
  4. cd ~/RUBRICAPP_PRODUCTION/rubricapp/Cloud
  5. Set up your BackEndFlask/.env file — see Section 2
  6. ./syscontrol.sh --init — enter your domain, answer Y to continue, answer Y to the SSL question
  7. ./syscontrol.sh --serve — enter your domain when prompted
WARNING: --init completely wipes the database. Do not run it on a server with live data unless you intend to reset it.

2 Environment Setup (.env)

Before running --init, the backend .env file must exist at: 

~/RUBRICAPP_PRODUCTION/rubricapp/BackEndFlask/.env

The script will fail during configure_db if this file is missing or has no database credentials. The .env file is not stored in the repo for security reasons. Contact the server admin or refer to the server guide for the correct values for each environment.

Required variables:

VariableDescription
FRONT_END_URLURL of the frontend (e.g. https://skill-builder.net)
SUPER_ADMIN_PASSWORDPassword for the super admin account
DEMO_ADMIN_PASSWORDPassword for demo admin account
DEMO_TA_INSTRUCTOR_PASSWORDPassword for demo TA account
DEMO_STUDENT_PASSWORDPassword for demo student account
SECRET_KEYFlask secret key for session signing
MYSQL_HOSTDatabase host (use localhost for local MySQL, or the managed DB host)
MYSQL_PORTDatabase port (3306 for local, 25060 for DO managed DB)
MYSQL_USERDatabase username
MYSQL_PASSWORDDatabase password
MYSQL_DATABASEDatabase name (typically rubricapp)
MYSQL_SSLMODESet to REQUIRED for managed DB, omit for local MySQL

Local MySQL example (test server):

FRONT_END_URL=http://localhost:3000
SUPER_ADMIN_PASSWORD=@super_admin_password123
DEMO_ADMIN_PASSWORD=demo_admin
DEMO_TA_INSTRUCTOR_PASSWORD=demo_ta
DEMO_STUDENT_PASSWORD=demo_student
SECRET_KEY=yoursecretkeyhere
MYSQL_HOST=localhost
MYSQL_PORT=3306
MYSQL_USER=rubricapp_admin
MYSQL_PASSWORD=yourpassword
MYSQL_DATABASE=rubricapp
Note: For the DO managed database or AWS RDS credentials, refer to the server guide. Do not commit .env files to the repo.

3 First Time Setup

Clone the repo onto the server and cd into it.

git clone https://github.com/Lunatic-Labs/rubricapp.git
cd ./rubricapp/Cloud/

Run syscontrol.sh with the --fresh option.

./syscontrol.sh --fresh

This creates a production directory at /home/$USER/RUBRICAPP_PRODUCTION/ and copies the repo into it. The original cloned directory is no longer needed after this step.

Navigate to the production directory:

cd /home/$USER/RUBRICAPP_PRODUCTION/rubricapp/Cloud/

Set up your .env file before continuing — see Section 2.

Then run:

./syscontrol.sh --init

You will be prompted for a domain name and asked whether the domain is already pointing to this server. See Section 4.3 and Section 5 for details.

Finally, start the application:

./syscontrol.sh --serve

4 Available Options

4.1 Help

Usage:

./syscontrol.sh --help

Displays all available options with short descriptions. Options are listed in logical order for setting up a new server.

4.2 Fresh

Usage:

./syscontrol.sh --fresh

Sets up the production directory structure at /home/$USER/RUBRICAPP_PRODUCTION/. Copies the repo from the current directory into the production folder. This is always the first command to run on a brand new server.

Note: After running --fresh, always re-run the script from the new location: /home/$USER/RUBRICAPP_PRODUCTION/rubricapp/Cloud/syscontrol.sh

4.3 Init

Usage:

./syscontrol.sh --init

Intended to be run once after --fresh. Runs the full setup in this order:

  1. Prompts for the domain name
  2. Asks if the domain is already pointing to this server
  3. Runs --install (all dependencies)
  4. Runs --configure if domain is ready, or --configure-no-ssl if not
  5. Runs dbcreate.py to initialize the database
WARNING: This option completely wipes the database and sets it up fresh. You will be prompted to confirm with Y before anything is deleted. Type n to abort safely. Never run this on a server with live production data.

4.4 Install

Usage:

./syscontrol.sh --install

Installs all dependencies needed to run the application:

  • System packages via apt — python3, nginx, redis-server, certbot, ufw, and others (see DEPS block in script for full list with explanations)
  • Python packages via pip — installed from BackEndFlask/requirements.txt inside the virtual environment at ~/RUBRICAPP_PRODUCTION/rubricapp-env/
  • Node.js via NVM — installs Node v20.11.1 and npm packages from FrontEndReact/package.json

4.5 Configure (SSL)

Usage:

./syscontrol.sh --configure

Configures all services with SSL. Runs the following steps in order:

SSL Certificate (Certbot)
Obtains a Let's Encrypt SSL certificate for the domain using Certbot in standalone mode. Temporarily stops Nginx and frees ports 80 and 443 for domain verification. If a certificate already exists it is skipped.

Gunicorn Service
Creates the systemd service at /etc/systemd/system/rubricapp.service. Runs the Flask backend via Gunicorn with 3 workers bound to a Unix socket at BackEndFlask/rubricapp.sock.

Nginx
Writes a single combined config to /etc/nginx/sites-available/rubricapp with:

  • Port 5000 SSL — proxies to the Gunicorn socket
  • Port 80 — redirects all HTTP traffic to HTTPS
  • Port 443 SSL — proxies to the React frontend on port 3000

Firewall (UFW)
Opens ports: 22 (SSH), 80 (HTTP), 443 (HTTPS), 5000 (backend API), 3000 (frontend)

Note: The domain must be pointing to this server's IP before running --configure, otherwise Certbot will fail. If the domain is not ready yet, use --configure-no-ssl instead.

4.6 Configure No SSL

Usage:

./syscontrol.sh --configure-no-ssl

Configures all services without SSL. Use this for test servers or any server where the domain is not yet pointing to the server's IP. Skips the Certbot step entirely. Runs the following steps:

Gunicorn Service
Same as --configure — creates /etc/systemd/system/rubricapp.service.

Nginx (HTTP only)
Writes a single config to /etc/nginx/sites-available/rubricapp with:

  • Port 80 — proxies to the React frontend on port 3000
  • Port 5000 — proxies to the Gunicorn socket (no SSL)

Firewall (UFW)
Same as --configure.

Note: To upgrade to SSL later, point the domain to the server then run --configure. It will overwrite the no-SSL Nginx config with the SSL version and obtain the certificate automatically. See Section 6.

4.7 Serve

Usage:

./syscontrol.sh --serve

Starts all services and makes rubricapp live. Stops any existing processes first for a clean start, then brings everything up in this order:

  1. Kills any existing processes
  2. Starts redis-server
  3. Starts the Gunicorn backend (rubricapp.service)
  4. Starts and reloads Nginx
  5. Builds the React frontend (npm run build)
  6. Starts the frontend as a background process on port 3000 (logs to FrontEndReact/frontend.log)

If you deploy new code, run --serve again to rebuild and restart everything. The frontend build step can take a minute.

Note: The frontend runs as a background process via nohup npm start. To check frontend logs: cat ~/RUBRICAPP_PRODUCTION/rubricapp/FrontEndReact/frontend.log

4.8 Update

Usage:

./syscontrol.sh --update

Stops all services and pulls the latest code from the remote repo. Does not automatically restart services — run --serve manually afterward to bring everything back up.

  1. Kills all running processes and services
  2. Fetches from remote and pulls if the local branch is behind origin/master

4.9 Status

Usage:

./syscontrol.sh --status

Prints the current status of all services and shows which processes are listening on each relevant port. Useful for a quick health check without restarting anything. Does not require a domain prompt.

Services checked:

  • nginx.service
  • rubricapp.service (Gunicorn backend)
  • redis-server.service

Ports checked: 5001, 5000, 3000, 80

4.10 Kill

Usage:

./syscontrol.sh --kill

Stops all services and cleans up processes. Also clears Python __pycache__ directories and .pyc files from the project for a clean state before the next deploy. Does not require a domain prompt.

Services stopped in order:

  • redis-server.service
  • rubricapp.service
  • nginx.service

Also kills any remaining processes on ports 5000, 5001, and 3000.

5 Domain Prompt

The script prompts for a domain name at startup for commands that need it (--init, --configure, --configure-no-ssl, --serve). Nothing is hardcoded — enter the correct domain for the server you are setting up.

ServerDomain
DO Productionskill-builder.net
DO Testskill-builder-testing.net
AWS Productionskill-builder.net
AWS Test(see server guide)
Note: Commands like --status and --kill do not prompt for a domain since they only manage running services.

6 Switching from No-SSL to SSL

If a server was set up with --configure-no-ssl and the domain is now pointing to the server, upgrading to SSL is a single command:

  1. Make sure the domain A record is pointing to this server's IP and has propagated
  2. Verify propagation: dig your-domain.com — the IP in the output should match the server
  3. Run: 
    ./syscontrol.sh --configure
  4. This will obtain the SSL cert, overwrite the no-SSL Nginx config with the SSL version, and reload Nginx
  5. Run --serve to restart everything cleanly
Note: Port 80 and 443 must be free when Certbot runs. The script handles this automatically by stopping Nginx before attempting certificate generation.
