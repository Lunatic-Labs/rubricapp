def get_test_student_token(course_id):
    try:
        # Get the admin making the request
        admin_id = get_jwt_identity()
        
        # Verify the course exists
        course = get_course(course_id)
        if not course:
            return jsonify({
                "success": False,
                "error": f"Course {course_id} not found"
            }), 404
        
        # Create unique email for test student tied to this course
        test_email = f"teststudent{course_id}@skillbuilder.edu"
        
        # Check if test student already exists
        test_student = User.query.filter_by(email=test_email).first()
        
        if not test_student:
            # Create new test student
            test_student = User(
                first_name="Test",
                last_name="Student",
                email=test_email,
                password="TestPassword123!",
                owner_id=course.admin_id if hasattr(course, 'admin_id') else admin_id,
                has_set_password=True,
                is_admin=False,
                consent=True,
                lms_id=None,
                reset_code=None
            )
            
            db.session.add(test_student)
            db.session.commit()
            
            # Refresh to get the user_id
            test_student = User.query.filter_by(email=test_email).first()
            
            if test_student:
                # Check if already enrolled
                existing = UserCourse.query.filter_by(
                    user_id=test_student.user_id,
                    course_id=course_id
                ).first()
                
                if not existing:
                    # Enroll test student in course with TestStudent role
                    test_user_course = UserCourse(
                        user_id=test_student.user_id,
                        course_id=course_id,
                        role_id=6,  # TestStudent role
                        active=True
                    )
                    db.session.add(test_user_course)
                    db.session.commit()
            else:
                return jsonify({
                    "success": False,
                    "error": "Failed to create test student"
                }), 500
        else:
            # Test student exists, ensure they're enrolled
            existing = UserCourse.query.filter_by(
                user_id=test_student.user_id,
                course_id=course_id
            ).first()
            
            if not existing:
                test_user_course = UserCourse(
                    user_id=test_student.user_id,
                    course_id=course_id,
                    role_id=6,
                    active=True
                )
                db.session.add(test_user_course)
                db.session.commit()
        
        # Generate JWT tokens for the test student
        access_token = create_access_token(identity=str(test_student.user_id))
        refresh_token = create_refresh_token(identity=str(test_student.user_id))
        
        # Return test student credentials
        response_data = {
            "success": True,
            "user": {
                "user_id": test_student.user_id,
                "user_name": f"{test_student.first_name} {test_student.last_name}",
                "first_name": test_student.first_name,
                "last_name": test_student.last_name,
                "email": test_student.email,
                "isAdmin": False,
                "isSuperAdmin": False,
                "has_set_password": True,
                "role_id": 6
            },
            "access_token": access_token,
            "refresh_token": refresh_token
        }
        
        return jsonify(response_data), 200
        
    except Exception as e:
        db.session.rollback()
        return jsonify({
            "success": False,
            "error": str(e)
        }), 500

View as Student Feature — Technical Documentation
Last Updated: April 6, 2026


1. Feature Overview
The View as Student feature allows administrators to switch their view to a test student perspective within a specific course. This enables admins to verify the student experience — including assessment tasks, completed assessments, teams, and skill development charts — without needing a separate student account or browser session.
Key Capabilities
• Admin can switch to a test student view from any course's Roster Dashboard
• A dedicated test student account is created per course in the backend (e.g., teststudent1@skillbuilder.edu for course 1)
• Admin credentials are preserved in sessionStorage for seamless switching back
• The student view displays a prominent "Switch Back to Admin" banner
• Spam protection prevents multiple rapid clicks during the switching process


2. Architecture & Data Flow
2.1 Switching to Student View (Admin → Student)
Trigger: Admin clicks the "View as Student" button on the RosterDashboard page.
Flow:
1. Save Admin Credentials — Admin's user, access_token, and refresh_token cookies are saved to sessionStorage under the key adminCredentials.
2. Fetch Test Student Token — A GET request is made to /api/courses/{course_id}/test_student_token. The backend either finds or creates a test student for that course and returns a JWT token pair and user object.
3. Set Student Cookies — The admin's cookies are cleared and replaced with the test student's access_token, refresh_token, and user object. The user object includes additional flags:
• viewingAsStudent: true
• originalCourseId: {course_id}
• viewingCourseName: {course_name}
4. Store Course Info — Course data is saved to sessionStorage under chosenCourse with role_id: 6 (TestStudent role).
5. Reload Page — window.location.reload() is called to re-initialize the app with the new credentials.
2.2 Switching Back to Admin (Student → Admin)
Trigger: Student clicks the "Switch Back to Admin" button in the StudentDashboard alert banner.
Flow:
1. Retrieve Admin Credentials — The saved admin credentials are read from sessionStorage key adminCredentials.
2. Blacklist Test Student Tokens — A POST request is made to /api/logout to invalidate the test student's tokens.
3. Restore Admin Cookies — The test student cookies are cleared and replaced with the original admin cookies.
4. Clean Up Session Storage — adminCredentials, chosenCourse, and testStudentCourse are removed from sessionStorage.
5. Reload Page — window.location.reload() re-initializes the app as the admin.


3. Backend Endpoint
GET /api/courses/{course_id}/test_student_token
Authentication: Requires a valid JWT (@jwt_required()). Only admins should access this endpoint.
Behavior:
• Looks up the test student by email pattern: teststudent{course_id}@skillbuilder.edu
• If the test student does not exist, creates one with:
• first_name: "Test"
• last_name: "Student"
• email: teststudent{course_id}@skillbuilder.edu
• owner_id: The admin's user ID
• Enrolls the test student in the course with role_id: 6 (TestStudent) in the UserCourse table
• Returns a JSON response containing access_token, refresh_token, and user object
Response Format:
{
  "access_token": "jwt_token_string",
  "refresh_token": "jwt_refresh_string",
  "user": {
    "user_id": 999,
    "user_name": "Test Student",
    "email": "teststudent1@skillbuilder.edu",
    "first_name": "Test",
    "last_name": "Student",
    "role_id": 6,
    "has_set_password": true
  }
}
Database Requirement: TestStudent Role
The role table must contain an entry for role_id: 6 with role_name: "TestStudent". Without this, the UserCourse insert will fail with a foreign key constraint error:
(pymysql.err.IntegrityError) (1452, 'Cannot add or update a child row:
a foreign key constraint fails (`rubricapp`.`usercourse`,
CONSTRAINT `usercourse_ibfk_3` FOREIGN KEY (`role_id`) REFERENCES `role` (`role_id`))')
Migration to add the role:
INSERT INTO role (role_id, role_name) VALUES (6, 'TestStudent');

Switch to Student Fails
• Admin credentials are restored from sessionStorage
• Partial test student cookies are cleared
• sessionStorage is cleaned up

• An alert is shown: "Error switching back to admin view"
• The isSwitchingBack flag is reset

Test Mock Requirements
The test file mocks global.fetch to handle all API endpoints. Key considerations for the mock:
utility.ts Resource Lookup Pattern:
state[dest || resource] = result.content[resource][0];
• The resource parameter (2nd argument to genericResourceGET) is used as the key to look up in result.content
• The value at that key must be an array, and [0] is extracted
• For array data (courses, users): content: { courses: [theArray] }
• For single objects (roles): content: { roles: [theObject] }
Endpoints requiring both singular and plural keys:
• /course → content.course and content.courses and content.course_count
• /role → content.role and content.roles
• /team_by_user → content.teams (resource name differs from URL path)
• /checkin → content.checkin (singular)
URL matching order matters — more specific patterns must come before general ones:
1. /courses/{id}/test_student_token (before /course)
2. /completed_assessment (before /assessment_task)
3. /team_by_user (before /team)
4. /course (general, last)


11. Known Issues & Considerations
Double API Path in Logout URL
The StudentDashboard.handleSwitchBack method constructs the logout URL as:
`${apiUrl}/api/logout?user_id=${adminCredentials.user.user_id}`
Where apiUrl = process.env.REACT_APP_API_URL which already contains /api. This results in a double path: http://127.0.0.1:5000/api/api/logout. The logout still works because the mock catches any URL containing /logout, but this should be fixed in production code.
Test Student Cleanup
Each course creates a persistent test student in the database. Consider implementing a periodic cleanup job to remove old test student accounts, or adding a cleanup endpoint that admins can trigger.
Test Student Isolation
Test students are isolated per course (e.g., teststudent1@skillbuilder.edu for course 1, teststudent2@skillbuilder.edu for course 2). This ensures that viewing as a student in one course does not affect another course's test data.
Password Security
The test student password is set during creation in the backend. It does not need to be known by users since authentication is handled via the token endpoint, not direct login.
